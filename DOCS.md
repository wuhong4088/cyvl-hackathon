## 1. Project Overview

SafeRoute AI is a climate-aware urban infrastructure risk tool built for the **Cyvl Physical AI Hackathon**. It ingests LiDAR point-cloud data from the Cyvl SDK (Somerville, MA), combines it with real historical weather from Open-Meteo, and produces an interactive priority map showing which pedestrian segments are most at risk from:

- ❄️ **Winter** — Black ice, icing, and slippery surfaces
- ☀️ **Summer** — Heat stress, shade deficit, and pavement degradation

The tool also integrates an **Autodesk Viewer** panel that renders a 3D mesh of the LiDAR scan directly in the browser, allowing users to visually evaluate spatial interventions (e.g., planting a tree or installing a bus shelter).

## 3. Data: Site Configuration & GeoJSON Files

All data produced by the processing pipeline is stored in `data/download/`.

### 3.1 Raw Inputs

| File | Description |
|---|---|
| `drive-download-*.zip` | Raw Cyvl export ZIP containing the `aboveGroundAssets.geojson` and two pavement score shapefiles |
| `global_xyz_rgb_icgu_76_12000_14000.laz` | LiDAR point cloud in the LAZ format (compressed LAS). Covers a tile of Somerville centered around the hackathon area. Used as input for the Autodesk 3D mesh pipeline. |

### 3.2 Processed GeoJSON Layers

These files are generated by `scripts/process_data.py` and loaded directly by the web app:

| File | Type | Content |
|---|---|---|
| `trees.geojson` | FeatureCollection (Points) | All `TREE` assets detected by the Cyvl LiDAR scan. Used to compute shade scores. 2,357 trees. |
| `ramps.geojson` | FeatureCollection (Points) | ADA-compliant `RAMP` assets. Used to compute physical feasibility scores. 748 ramps. |
| `obstacles.geojson` | FeatureCollection (Points) | Infrastructure obstacles: utility poles, traffic signal poles, fire hydrants, luminaires, catch basins. 1,996 total. |
| `transit.geojson` | FeatureCollection (Points) | Pedestrian transit signals (`STAND_ALONE_PEDESTRIAN_HEAD`, `PEDESTRIAN_PUSH_BUTTON`). Proxy for pedestrian crossing exposure. 263 signals. |
| `sidewalks.geojson` | FeatureCollection (LineStrings) | Sidewalk and shared path segments. Each feature has `has_walk: true/false` and `sidewalk_type`. 864 segments. |
| `curbs.geojson` | FeatureCollection (LineStrings) | Curb geometries from the Cyvl scan. |
| `risk_map.geojson` | FeatureCollection (Points) | One point per pavement segment. Contains both `winter_risk` and `summer_risk` scores (0–1) plus all contributing factor fields. 894 segments. |

### 3.3 JSON Data Files

#### `weather.json`
Historical weather data fetched from the [Open-Meteo Archive API](https://archive-api.open-meteo.com) for two reference periods:

- **Winter:** Nov 17–24, 2025 (matches the Cyvl scan period in Somerville)
- **Summer:** Jul 15–22, 2025 (representative heat-stress week)

Each period contains daily rollups of temperature, humidity, precipitation, snowfall, and wind speed, plus a computed `risk` score (0–1) per day. The `overall_risk` average for each period is used as the **climate severity** factor in the risk model.

#### `summary.json`
A pre-computed summary written after the risk pipeline completes. Contains:

- `location`, `total_segments`, `asset_counts` — summary statistics
- `weather` — seasonal risk averages and periods
- `winter` / `summer` — distribution across CRITICAL / HIGH / MEDIUM / LOW categories, and the top-5 highest-risk streets
- `selected_sites` — two hand-picked demonstration sites chosen by the algorithm:
  - **Summer site:** "Hot shade-poor sidewalk corridor" on Street1953 (risk: 0.697, HIGH)
  - **Winter site:** "Icy salt-priority transit crossing" on Elm Street (risk: 0.897, CRITICAL)

#### `autodesk_config.json`
Written automatically by `autodesk/full_stack_autodesk_creation.py` after a successful APS upload and translation. Contains:

```json
{
  "urn": "<base64-encoded APS object URN>",
  "token": "<APS access token (expires in 1 hour)>",
  "center": [<X>, <Y>, <Z>]   // centroid of the point cloud in UTM coordinates
}
```

This config is loaded by `web/app.js` at startup to initialize the Autodesk Forge Viewer embedded in the 3D Spatial Planner panel.

> ⚠️ **Note:** The `token` field expires after 1 hour. If the viewer shows a blank screen or fails to load the model, you need to re-run `autodesk/full_stack_autodesk_creation.py` to obtain a fresh token.

## 4. Scripts: Processing Pipeline (`process_data.py`)

**File:** [`scripts/process_data.py`](scripts/process_data.py)  
**Run with:** `python3 scripts/process_data.py` (from project root)  
**Output:** All GeoJSON and JSON files in `data/download/`

This is the main ETL (Extract, Transform, Load) pipeline. It has no external web dependencies except for Open-Meteo weather fetching. Here is what each stage does:

---

### Stage 1 — Load Assets from ZIP (`load_assets_from_zip`)

Opens the raw Cyvl ZIP export and reads `CityofSomervilleMAMarketingDemo-aboveGroundAssets.geojson` directly from the archive into memory. No intermediate extraction to disk.

---

### Stage 2 — Process Assets (`process_assets`)

Iterates all GeoJSON features from the above-ground assets file and categorizes them by `asset_type`:

| Asset Type | Category | Output File |
|---|---|---|
| `TREE` | Point | `trees.geojson` |
| `RAMP` | Point | `ramps.geojson` |
| `UTILITY_POLE`, `TRAFFIC_SIGNAL_POLE`, `HYDRANT`, `LUMINARIES`, `CATCH_BASIN` | Point | `obstacles.geojson` |
| `STAND_ALONE_PEDESTRIAN_HEAD`, `PEDESTRIAN_PUSH_BUTTON` | Point | `transit.geojson` |
| `SIDEWALK` | LineString | `sidewalks.geojson` |
| `CURB` | LineString | `curbs.geojson` |

Features are filtered to a bounding box (`LON: -72 to -70`, `LAT: 42 to 43`) to reject any stray out-of-region coordinates. For sidewalks, the `has_walk` property is set to `true` only when `sidewalk_type` is `"Sidewalk"` or `"Shared Path"`.

**Key coordinate collections returned** (used in risk computation):  
`tree_coords`, `ramp_coords`, `obstacle_coords`, `catch_coords`, `transit_coords`, `sidewalk_centroids`

---

### Stage 3 — Load Pavement Data (`load_pavements_from_zip`)

Reads two nested ZIP shapefiles from within the Cyvl archive:

1. **`30ft Pavement Scores.zip`** — provides `client_seg → address` mapping
2. **`Segment-to-Segment Pavement Scores.zip`** — provides PCI (Pavement Condition Index) scores per segment

Segments without an address are resolved via **spatial nearest-neighbour search**: the script finds the closest addressed segment within 250 metres and inherits its street name.

Returns a list of dicts, each with: `lon`, `lat`, `address`, `pci_score`, `label`, `length_ft`.

---

### Stage 4 — Fetch Weather (`fetch_weather`)

Calls the Open-Meteo Archive API for hourly temperature, humidity, precipitation, snowfall, and wind speed for both the winter and summer reference periods. On network failure, falls back to hardcoded representative values (defined in `_winter_fallback` and `_summer_fallback`).

Daily rollups are computed from the hourly data. A `risk` score is computed per day using a rule-based formula:

- **Winter risk:** Adds weight for sub-freezing temperatures, precipitation, snowfall, and high humidity
- **Summer risk:** Adds weight for high max temperature, high humidity, and low wind speed

Output: `data/download/weather.json`

---

### Stage 5 — Compute Risk (`compute_risk`)

The core scoring engine. For each pavement segment, the algorithm:

1. Counts nearby assets within **60 metre radius** (using Haversine distance):
   - `n_trees`, `n_transit`, `n_ramps`, `n_catch` (catch basins), `n_walk` (walkable sidewalks)

2. Derives intermediate scores:
   - `shade_score = min(1.0, n_trees / 8)` — 8 trees = fully shaded
   - `shade_deficit = 1 - shade_score`
   - `pavement_risk = 1 - (pci / 100)` — higher PCI = better pavement
   - `drainage_deficit = 1 - min(1.0, n_catch / 3)`
   - `ped_exposure = min(1.0, n_walk / 4)`
   - `transit_exposure = min(1.0, n_transit / 3)`
   - `feasibility = (ped_exposure + ramp_exposure) / 2`

3. Computes seasonal risk scores using a **weighted 5-factor model**:

| Factor | Weight | Winter Meaning | Summer Meaning |
|---|---|---|---|
| Climate severity | 30% | Sub-zero temperatures + precip | Max heat + humidity |
| Surface condition | 25% | Ice retention (shade + drainage) | Heat amplification (shade deficit + bad pavement) |
| Pedestrian exposure | 20% | Foot traffic density | Foot traffic density |
| Transit exposure | 15% | Crossing hazard (signals nearby) | Crossing hazard (signals nearby) |
| Physical feasibility | 10% | Ease of intervention | Ease of intervention |

4. Labels each segment as `CRITICAL` (≥0.75), `HIGH` (≥0.55), `MEDIUM` (≥0.35), or `LOW`.

Output: `data/download/risk_map.geojson`

---

### Stage 6 — Summary Stats (`_summary_stats`)

Selects two representative demo sites:
- **Summer site:** Highest summer-risk segment that has at least one nearby sidewalk (pedestrian-facing heat corridor)
- **Winter site:** Highest winter-risk segment that has at least one nearby transit signal (high-traffic icy crossing)

Computes distribution counts and top-5 ranked addresses for each season.

Output: `data/download/summary.json`

## 5. Autodesk: 3D Mesh Pipeline (`full_stack_autodesk_creation.py`)

**File:** [`autodesk/full_stack_autodesk_creation.py`](autodesk/full_stack_autodesk_creation.py)  
**Run with:** `python3 autodesk/full_stack_autodesk_creation.py` (from project root)  
**Dependencies:** See [`autodesk/requirements.txt`](autodesk/requirements.txt)

This is a standalone pipeline that converts the raw `.laz` point cloud into a 3D mesh, uploads it to Autodesk Platform Services (APS / Forge), and produces the `autodesk_config.json` file used by the web app.

### Key Dependencies

| Package | Purpose |
|---|---|
| `laspy` | Reads LiDAR `.laz` / `.las` point cloud files |
| `open3d` | 3D geometry processing, voxel downsampling, normal estimation, Poisson reconstruction |
| `numpy` | Point array manipulation |
| `requests` | HTTP calls to the APS REST API |

---

### Pipeline Steps

#### Step 0 — `generate_obj_from_laz` — LiDAR → OBJ Mesh

1. **Reads** the `.laz` file using `laspy`. The full file is ~352 MB.
2. **Stride sampling:** Takes every 200th point to reduce initial load (`sample_stride=200`).
3. **Centers** the point cloud to the origin (subtracts centroid).
4. **Voxel downsampling:** Uses `open3d.voxel_down_sample` starting at voxel size `0.2m`, increasing by `0.05m` until the point count drops below 350,000. This prevents memory overload while preserving detail.
5. **Normal estimation:** Uses KNN with k=25 neighbours, then orients normals consistently across the surface with `orient_normals_consistent_tangent_plane(100)`.
6. **Poisson Surface Reconstruction:** Runs at `depth=9`, producing a watertight mesh from the oriented point cloud. Higher depth = sharper geometric detail.
7. **Artifact removal:** Removes the bottom 5% lowest-density vertices to clean up noise at the mesh boundary.
8. **Exports** the clean mesh to `autodesk/global_xyz_rgb_icgu_76_12000_14000.obj`.

#### Step 1 — `get_access_token` — APS OAuth

Authenticates with Autodesk Platform Services using client credentials (2-legged OAuth). Requests scopes: `bucket:create bucket:read data:create data:write data:read`. Returns a Bearer token valid for 1 hour.

> Credentials are loaded from environment variables `APS_CLIENT_ID` and `APS_CLIENT_SECRET`.

#### Step 2 — `upload_file` — Upload OBJ to OSS (S3)

Uses the APS S3-accelerated upload path:
1. Requests a signed S3 upload URL from APS.
2. Streams the `.obj` file directly to AWS S3.
3. Finalizes the upload with APS to register the object in the OSS bucket.
4. Base64-encodes the resulting `objectId` to produce the APS URN.

#### Step 3 — `translate_file` — Convert to SVF2

Submits a translation job to the APS Model Derivative API, requesting `svf2` format (cloud-streaming) with both 2D and 3D views.

#### Step 4 — `check_translation_status` — Poll Until Complete

Polls the APS manifest endpoint every 5 seconds until status is `"success"` or `"failed"`.

#### Step 5 — `generate_html_file` + `start_local_server`

Writes a standalone `index.html` (Forge Viewer) and launches a `python3 -m http.server` for quick local preview.

Additionally, writes `data/download/autodesk_config.json` containing the URN, token, and point-cloud center — consumed by the main web app.

### 6.1 `web/index.html` — App Shell

The HTML file is a declarative shell; all interactive logic lives in `app.js`. Key structural sections:

#### `<head>`
- Loads **MapLibre GL JS 4.7.1** (open-source Mapbox alternative, no API token required)
- Loads **Autodesk Viewer 7.x** CSS + JS (from Autodesk CDN)
- Loads **Chart.js 4.4.3** (for the weather bar chart in the sidebar)
- Loads **Google Fonts**: `Inter` (UI) and `JetBrains Mono` (monospace accents)
- Links `style.css`

#### `<header class="topbar">`
Fixed top navigation bar. Contains:
- **Brand** — SafeRoute AI logo + subtitle
- **Season switcher** — Two buttons (`❄️ Winter` / `☀️ Summer`) that switch the entire map between risk modes
- **Stats pills** — Live-updating counts for Trees, Segments, Critical zones
- **Weather widget** — Shows the season icon, period label, and average temperature

#### `<aside class="sidebar">`
Left-side collapsible panel. On mobile, this is a bottom sheet with drag-to-expand behavior. Sections inside:

| Section | Content |
|---|---|
| **Climate Stress Overview** | Semi-circle gauge (canvas), 4-bar risk distribution (CRITICAL / HIGH / MEDIUM / LOW) |
| **Selected Priority Sites** | Two demo sites from `summary.json`, click to fly to location |
| **5-Factor Priority Score** | Formula explainer with weights for the current season |
| **Weather** | Chart.js bar chart of daily risk score for the active season's week |
| **High-Priority Sites** | Top-5 ranked streets for the active season |
| **Map Layers** | Toggle switches for: Risk Heatmap, Critical Zones, Trees, Sidewalks, Crossings, Obstacles, ADA Ramps |

#### `<main class="map-container">`
Full-viewport map area. Contains:

- **`#map`** — MapLibre GL canvas target
- **`#autodesk-viewer-panel`** — Slide-in right panel for 3D spatial planning. Opens when a segment is clicked on the map. Contains:
  - Autodesk Forge Viewer (`#forgeViewer`)
  - **Spatial Intervention selector:** Three buttons — `None`, `🌳 Plant Tree (Natural)`, `🚏 Bus Shelter (Artificial)` — which overlay procedural 3D geometry onto the mesh
  - **Municipal Regulation Audit checklist** — Dynamically generated checklist based on selected intervention and site context
- **Map Legend** — Color key for risk levels and asset types
- **Loading overlay** — Shown while GeoJSON files are fetching

#### `<template id="popup-tpl">`
A hidden HTML template for the map click popup. Cloned and populated dynamically with segment data (address, risk score, season-specific factors).

### 6.3 `web/app.js` — Application Logic

The main JavaScript module (~1,050 lines). All logic is imperative vanilla JS (no framework).

#### Key Modules / Functions

| Function | Purpose |
|---|---|
| `initMap()` | Initializes MapLibre map centered on Somerville, MA (42.39°N, 71.10°W, zoom 13.5) with dark-matter CartoCDN basemap |
| `loadData()` | `Promise.all` fetch of all 7 GeoJSON + 2 JSON + `autodesk_config.json` files |
| `addMapLayers()` | Registers all MapLibre sources and layers: heatmap, risk circles, sidewalks, obstacles, ramps, transit, trees, critical pulsing rings |
| `setSeason(s)` | Switches the active season, re-paints all data-driven map styles, updates sidebar text, re-renders chart and stats |
| `bindSeasonSwitch()` | Event listeners on the top-bar season buttons |
| `bindLayerToggles()` | Checkbox listeners that set MapLibre layer visibility |
| `bindMapEvents()` | Click handler on `risk-circles` → shows popup, opens Autodesk panel for CRITICAL/HIGH sites |
| `populateStaticSidebar()` | Renders pills, selected sites cards, and top-5 site list from `summary.json` |
| `renderWeatherChart(season)` | Builds a Chart.js bar chart of daily risk for the active season |
| `renderGauge(avgRisk)` | Draws a semicircle gauge on the `<canvas id="riskGauge">` using the 2D Canvas API |
| `openAutodeskPanel(feature)` | Slides in the right panel; loads or re-uses the Autodesk Forge Viewer; calls `loadForgeModel()` |
| `loadForgeModel(urn, token)` | Initializes `Autodesk.Viewing.GuiViewer3D`, loads the OBJ model via SVF2 URN |
| `applyIntervention(type, feature)` | Called when an intervention button is clicked. Overlays a procedurally generated `THREE.js` mesh (tree cylinder or shelter box) on the Forge Viewer's scene. Calls `runRegulationAudit()`. |
| `runRegulationAudit(type, feature)` | Evaluates the selected intervention against segment data (nearby ramps, sidewalk width proxy, transit signals). Renders a checklist of ✅/❌ results in the panel. |
| `buildPopup(feature)` | Clones `<template id="popup-tpl">`, populates with season-aware risk score and 5-factor breakdown, returns HTML string |

#### Season-Aware Data Flow

All risk visualization is driven by `currentSeason` (`'winter'` or `'summer'`). When `setSeason()` is called:
1. MapLibre paint properties are updated: `['get', 'winter_risk']` ↔ `['get', 'summer_risk']`
2. The critical-zones source is rebuilt (`criticalFC()`)
3. The sidebar gauge, bars, and chart are re-rendered
4. The legend title and weather widget update

#### Autodesk Integration

The Forge Viewer is loaded lazily when a map segment is clicked. The `data/download/autodesk_config.json` file provides the `urn` and `token`. The viewer is initialized once and reused across segment clicks. When an intervention is applied, `THREE.Mesh` objects are injected directly into the viewer's `impl.scene` (the underlying Three.js scene that backs the Forge Viewer).

## 7. Running the Project Locally

### Prerequisites

- Python 3.12 (via pyenv or system install)
- pip packages for the processing script: `geopandas`
- pip packages for Autodesk pipeline: see `autodesk/requirements.txt`

### Step 1 — Install dependencies

```bash
# For the data pipeline
pip install geopandas

# For the Autodesk pipeline
pip install -r autodesk/requirements.txt
```

### Step 2 — Run the data processing pipeline

```bash
cd /path/to/cyvl-hackathon
python3 scripts/process_data.py
```

This will write all GeoJSON and JSON files to `data/download/`. It takes ~1–3 minutes (mostly weather fetch + pavement processing).

### Step 3 — (Optional) Generate the 3D Autodesk mesh

Set your APS credentials as environment variables:

```bash
export APS_CLIENT_ID="your_client_id"
export APS_CLIENT_SECRET="your_client_secret"
python3 autodesk/full_stack_autodesk_creation.py
```

This takes **30–60 minutes** (Poisson reconstruction is compute-intensive). It will write `autodesk_config.json` when complete.

> ⚠️ The APS token in `autodesk_config.json` expires after **1 hour**. If the 3D viewer shows a blank panel, re-run this script to get a fresh token.

### Step 4 — Serve the website

```bash
python3 -m http.server 8000 --directory /path/to/cyvl-hackathon
```

Then open: [http://localhost:8000/web/index.html](http://localhost:8000/web/index.html)
